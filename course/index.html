Codebase Course Inside SkinPair
React Vite TypeScript No Backend Local-First MVP

How SkinPair thinks before it tells you two products do not belong together.

This project is a browser-only skincare pairing analyzer. You paste ingredient lists, it detects key treatment tags, and it compares every saved product against a handwritten compatibility rulebook.

The goal of this course is not to turn you into an engineer. It is to give you enough technical fluency to steer an AI assistant, catch bad architecture suggestions, and debug this app without feeling lost.

What it does

Builds a tiny skincare shelf, detects ingredients, then warns about risky layering.

Why it is interesting

It feels like an app with “backend behavior,” but the whole thing lives in your browser.

What you will learn

State, persistence, rule engines, UI orchestration, and how to debug a local-first product.

01
Why you should care: this teaches you where one click turns into multiple code actions.

The moment you add a product, three jobs start at once.

Imagine a parcel entering a sorting depot. The product name, ingredient list, and shelf refresh are all separate jobs, but the user experiences them as one smooth moment. That is the core trick of this app.

What a user actually experiences

Step 1 Capture

You type a product name and paste ingredients into a simple form.

Step 2 Interpret

The app guesses product type and extracts meaningful tags from the ingredients.

Step 3 Judge

The analyzer checks every saved product against every other saved product.

Aha!

A lot of “AI-powered” products are really orchestration products. The hard part is not just a single clever function. It is coordinating multiple small steps so the app feels instant and trustworthy.

Trace the journey like a courier route

Click the route stops. Each stop shows one responsibility, which is exactly the vocabulary you want when telling an AI agent where to make changes.

Stop 1: turn raw text into a payload

The app trims whitespace and packages the two fields into a payload. This is the handoff format the rest of the app understands.

Stop 2: save into the shelf

The save call does not go to a server. It stores the new product in browser localStorage, adds an ID, and attaches detected tags.

Stop 3: recompute the whole shelf

After saving one product, the app immediately reruns the pairing analysis for every product. That means the analyzer behaves like a fresh judge, not like a patchwork of partial updates.

Stop 4: redraw the screen

Only after the data is ready does the UI replace the shelf list, clear the form, and show the new pairing cards. That sequencing is why the app feels deliberate instead of glitchy.

02
Why you should care: once you know the cast, you can tell AI exactly which file should own a change.

Meet the cast of characters.

Think of this codebase like a theater troupe. One actor faces the audience, one carries props, one reads the script, and one keeps the notebook of past performances.

Tap a character to see its job

App.tsx: the stage manager

It owns the screen, the form, the loading states, and the timing of refreshes. When you ask AI to “change the user flow,” this file is often the right stage.

  • Tracks form input and UI state
  • Decides when to save, refresh, and show errors
  • Renders shelf cards and pairing cards

shelfService.ts: the polite receptionist

This file looks like an API, even though everything stays local. It gives the UI simple verbs like fetch, create, and delete.

  • Hides storage details from the UI
  • Makes future backend upgrades easier
  • Keeps the interface layer simpler to read

localAnalysis.ts: the chemist

This is where raw ingredient text becomes detected tags, sorted products, and final pairing verdicts. If the analyzer feels “wrong,” this file is your prime suspect.

  • Normalizes strings
  • Detects tags and product type
  • Compares all product pairs against rules

localProductStore.ts: the shelf drawer

This file is the long-term drawer that stores products between refreshes. It assigns IDs and manages add/remove behavior.

  • Reads and writes browser localStorage
  • Owns the product list and next ID counter
  • Recovers from corrupt data by rebuilding when needed

localCache.ts: the quick snapshot notebook

This is the app’s “open to where we left off” notebook. It stores products, pairings, tags, and a save timestamp.

  • Supports faster first paint after refresh
  • Validates snapshot shape before trusting it
  • Stores a savedAt timestamp for sync messaging

pairingDataset.ts: the rulebook

This giant file is not logic. It is knowledge. It lists ingredient keywords, compatibility rules, conflict rules, and product-type hints.

  • Maps ingredient words to tags
  • Defines which tag pairs are safe or risky
  • Includes multilingual ingredient variants

The file tree in plain English

src/App.tsx

The one big React component that users actually see.

src/lib/shelfService.ts

A clean function layer so the UI does not need to know where data really lives.

src/lib/localAnalysis.ts

The analysis engine that turns words into tags, tags into reasons, and reasons into cards.

src/lib/localProductStore.ts

The durable shelf database, except the “database” is just browser storage.

src/lib/localCache.ts

The fast-open snapshot that makes refreshes feel friendlier.

src/lib/pairingDataset.ts

The catalog of tags, keyword detectors, and pairing judgments.

Switch scenarios to watch different message paths

Add product path

  1. Form fields become a payload.
  2. The service calls the local product store.
  3. The analysis engine reruns on the current shelf.
  4. The snapshot cache is saved.
  5. React state updates and the cards redraw.

Delete product path

  1. The UI marks one product as “currently deleting.”
  2. The service removes it from localStorage.
  3. The analyzer recalculates all remaining pairs.
  4. The snapshot is overwritten so refreshes match the new shelf.
  5. The deleting spinner disappears.

Refresh analysis path

  1. The UI flips into refresh mode.
  2. Products, pairings, and tags are fetched together.
  3. The app saves a fresh snapshot and timestamp.
  4. React updates products, tags, tag lookup, and cards in one transition.
  5. The loading banner disappears.

Code ↔ English: refresh three things together

Code 
try {
  const [nextProducts, nextPairings, nextTags] = await Promise.all([
    fetchProducts(),
    fetchPairings(),
    fetchTags(),
  ]);
  const nextSavedAt =
    saveLocalShelfSnapshot({
      products: nextProducts,
      pairings: nextPairings,
      tags: nextTags,
    }) ?? localSnapshotSavedAt;

  startTransition(() => {
    setProducts(nextProducts);
    setPairings(nextPairings);
    setTags(nextTags);
    setTagLookup(buildTagLookup(nextTags));
    setLocalSnapshotSavedAt(nextSavedAt);
  });
Plain English

Ask for products, pairings, and tags at the same time instead of one after another.

Save the newly gathered snapshot so refreshes can reopen the same scene.

If the save fails, keep the previous timestamp instead of crashing the flow.

Use a React transition so the UI can update a bundle of state changes together.

Rebuild the tag lookup map so labels stay fast and human-readable.

04
Why you should care: this explains the real “database” of the app and its tradeoffs.

The outside world is not a server. It is your browser drawer.

Picture a locked desk drawer with two notebooks inside. One notebook stores the durable shelf. The other notebook stores a convenient snapshot of what the app last showed you.

The three pieces of persistence

Notebook 1 Product store

Lives under the key skincare-local-product-store. It keeps the real list of products and the next numeric ID.

Notebook 2 Shelf snapshot

Lives under skincare-shelf-snapshot. It stores products, pairings, tags, and a timestamp so refreshes open fast.

Rulebook Static dataset

This is not saved by the user at all. It ships with the app bundle as code and comes back every load.

Aha!

Local-first software often uses more than one persistence layer. The trick is knowing which layer is authoritative and which one is just a speed shortcut.

Which drawer are you debugging?

Use this drawer when products disappear entirely

If a product is gone after refresh, the durable store is involved. This is the closest thing the app has to a database.

Use this drawer when the UI reopens stale information

If the app briefly shows old pairings or old tags before refreshing, the snapshot notebook is part of the story.

Use this drawer when the analyzer never learned a concept

If a product contains a new ingredient spelling and the app misses it every time, the rulebook needs more knowledge.

Code ↔ English: writing to the durable shelf

Code 
export const addStoredProduct = (payload: ProductInput) => {
  const store = loadStore();
  const createdProduct = buildProduct(store.nextId, payload);
  const nextStore = {
    version: PRODUCT_STORE_VERSION,
    nextId: store.nextId + 1,
    products: [...store.products, createdProduct],
  } satisfies ProductStore;

  saveStore(nextStore);
  return createdProduct;
};
Plain English

Open the current shelf drawer and read the latest stored data.

Turn the new form input into a finished product with an ID and analyzed tags.

Build the next version of the store with one higher ID and one more product.

Write that whole new store back into browser storage.

Return the created product so the UI can update immediately.

The clever trick is not magic. It is a rule engine with a broad vocabulary.

The best metaphor here is a matchmaker with a giant address book. The app does not “understand skincare” in a human sense. It recognizes names, groups them into tags, then checks those tags against a curated list of relationships.

Code ↔ English: keyword detection in the analyzer

Code 
pairingData.tagKeywords.forEach((keywordRule) => {
  const normalizedKeyword = normalizeText(keywordRule.keyword);
  if (!normalizedKeyword) {
    return;
  }

  const didMatch =
    keywordRule.match_type === "exact"
      ? exactTokens.has(normalizedKeyword)
      : normalizedIngredients.includes(normalizedKeyword);

  if (didMatch) {
    matchedTags.push(keywordRule.tag_code);
  }
});
Plain English

Walk through every known ingredient keyword in the dataset.

Normalize the keyword so accents, case, and formatting differences are less likely to break matching.

Use either strict token matching or looser “contains” matching depending on the rule.

If the product text matches, add that tag to the detected list.

Aha!

Rule engines often look deceptively simple. Their power comes from the quality of the catalog they read, not just the loop that iterates through it.

06
Why you should care: this is the escape hatch when an AI assistant loops or guesses.

When things break, debug by symptom, not by panic.

Think like a medical triage nurse. Do not inspect everything. Start from the symptom, pick the most likely layer, and move inward. That is how you break out of useless AI bug loops.

Pick a symptom, then inspect the first likely layer

Symptom “I added a product and it vanished on refresh.”

Start with localProductStore.ts. The durable shelf write is your first suspect.

Symptom “The shelf returns, but the warnings are outdated.”

Start with localCache.ts and the snapshot refresh path.

Symptom “This ingredient is never recognized.”

Start with pairingDataset.ts and detectTags.

Symptom “The card says the wrong explanation.”

Start with conflict and compatibility rules in the dataset, then the reason-selection logic.

The big picture you can now use in prompts

Prompt pattern Rule update

“Add a new compatibility rule in pairingDataset.ts and keep the analysis engine unchanged.”

Prompt pattern Storage fix

“Investigate why localCache.ts can restore stale pairings after refresh.”

Prompt pattern UI change

“Add a manual tag override in App.tsx and pass it through the product payload.”

Aha!

The architecture is now small enough to carry in your head: UI orchestration, service verbs, local persistence, and a dataset-driven rule engine. That mental model is how you become hard to mislead.

Code ↔ English: refuse bad snapshot data

Code 
if (
  parsedSnapshot.version !== LOCAL_SHELF_SNAPSHOT_VERSION ||
  !Array.isArray(parsedSnapshot.products) ||
  !isPairingAnalysis(parsedSnapshot.pairings) ||
  !Array.isArray(parsedSnapshot.tags) ||
  typeof parsedSnapshot.savedAt !== "string"
) {
  return null;
}
Plain English

Before trusting saved snapshot data, check that its version and shape still match what the app expects.

If products, tags, pairings, or the timestamp look wrong, reject the whole snapshot.

This is defensive programming: it is better to rebuild safely than to render nonsense from corrupted stored data.
